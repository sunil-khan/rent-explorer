# Rent Explorer

A location-intelligence tool for exploring Helsinki metro rental listings on an interactive map. Filter by rent, rooms, and property type; view price distributions by area via choropleth; and search for nearby listings by radius.

## Quick Start

**Prerequisites:** Docker and Docker Compose

```bash
git clone <repo-url>
cd rent-explorer
docker-compose up --build
```

The database seeds automatically on first startup (~850 listings, 12 areas).

- **Frontend:** http://localhost:5173
- **API:** http://localhost:8000
- **API Docs (Swagger):** http://localhost:8000/docs

## API Endpoints

| Endpoint             | Description                                                  | Key Params                                                                                   |
| -------------------- | ------------------------------------------------------------ | -------------------------------------------------------------------------------------------- |
| `GET /listings`      | Listings within a viewport with attribute filters            | `min_lng`, `min_lat`, `max_lng`, `max_lat`, `rent_min`, `rent_max`, `rooms`, `property_type` |
| `GET /areas/stats`   | GeoJSON FeatureCollection with median rent and €/m² per area | —                                                                                            |
| `GET /listings/near` | Listings within a radius sorted by distance                  | `lat`, `lng`, `radius_m`                                                                     |
| `GET /health`        | Health check                                                 | —                                                                                            |

Full interactive docs at `/docs` (auto-generated by FastAPI).

## Features

- **Map view** — listings as points on a MapLibre GL map
- **Choropleth** — areas colored by median €/m², with legend
- **Filters** — rent range, rooms, property type; updates map live
- **Listing detail** — click any point to see rent, size, €/m², rooms
- **Summary panel** — count and median €/m² of visible listings
- **Radius search** — click map to find listings within X meters
- **Value signal** — points colored green (good value) to red (above area median)
- **Debounced refetch** — listings update as you pan/zoom the map

## Non-Trivial Engineering Decision

**Query-time spatial join vs pre-computed area assignment.** When loading listings, we could pre-compute which area each listing belongs to and store `area_code` directly on the listing row. Instead, we compute `ST_Contains` at query time. Area boundaries are not immutable — districts get redrawn, boundaries corrected. Pre-computing creates a dual-write problem. With query-time joins, the next request automatically reflects updated boundaries. For 850 listings × 12 areas with GiST indexes, the join is sub-millisecond. At scale, a materialized view refreshed on schedule gives pre-computed read performance with query-time correctness.

See [docs/decisions.md](docs/decisions.md) for full architecture rationale.

## What I'd Do With More Time

- **Clustering/vector tiles** — at scale, use map clustering (Supercluster) for visual grouping; for larger datasets, switch to vector tiles (PostGIS ST_AsMVT) for efficient rendering
- **Caching** — HTTP cache headers for browser-level caching, Redis for shared server-side cache. `/areas/stats` is the prime candidate — same result for every user, changes only when listings are added
- **CI/CD** — GitHub Actions pipeline: lint, test, build, deploy
- **Auth** — API key or JWT for production use
- **Monitoring** — structured logging, request tracing, error alerting
- **E2E tests** — Playwright tests for critical user flows
- **Production Docker** — multi-stage frontend build with nginx, Gunicorn for API
- **Materialized views** — for `/areas/stats` at scale (pre-compute median aggregations, refresh on schedule)
